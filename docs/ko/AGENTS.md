# Agents 지침

## 개요

아키텍처 규칙의 1차 기준은 리포지토리 `skills/` 디렉터리입니다.
`skills/`는 `docs/` 없이도 동작하도록 자체 포함(self-contained)된 규칙 집합입니다.

## 스킬 기반 활성화

- `skills/fla-architecture/SKILL.md` (항상 적용)
- `skills/fla-file-directory-convention/SKILL.md` (파일/디렉터리 생성·이동·리네임, 모듈 구조 변경 시)
- `skills/fla-layer-resolver/SKILL.md` (경로의 가장 깊은 레이어 기준으로 유효 레이어 스킬 선택)

레이어 스킬:
- `/_pages/` -> `skills/fla-page/SKILL.md`
- `/_containers/` -> `skills/fla-container/SKILL.md`
- `/_states/` -> `skills/fla-state/SKILL.md`
- `/_components/` -> `skills/fla-component/SKILL.md`
- `/_apis/` -> `skills/fla-api/SKILL.md`
- `/_utils/` -> `skills/fla-util/SKILL.md`

## 작업 방침

수정 대상 파일의 기존 스타일/흐름을 먼저 파악하고 변경 범위를 최소화합니다.
레이어 디렉터리 밖을 수정할 때는 글로벌 규칙과 파일/디렉터리 규칙만 우선 적용합니다.

## 실행 정책 (요청 반영)

- 한 번에 하나의 함수만 구현합니다.
- 한 함수 단위를 진행할 때:
  1. 필요한 최소 인터페이스/타입을 먼저 정의합니다.
  2. 그 함수만 구현합니다.
  3. 입력/출력/상태 변경과 엣지 케이스를 한 번에 설명합니다.
- 필요한 경우가 아니면 광범위한 공통 타입/설정 파일을 미리 만들지 않습니다.
- 각 함수 단위 전에 보고합니다.
  - 함수명
  - 새로 추가된 인터페이스/타입(있을 경우)
  - 기대 동작
  - 검토 포인트
  - 되돌리기 포인트
- 각 함수 단위 후에 보고합니다.
  - 변경 내용
  - 확인 방법
  - 미해결 질문/전제 조건
- 변경은 가역적으로 제한된 범위에서 유지합니다.

## 학습 우선 워크플로우

1. 가장 작은 검증 가능한 동작부터 시작합니다.
   - 예: “Hello world”가 렌더되는지
2. 하나의 UI 상호작용(버튼 클릭/이벤트 바인딩)
3. 하나의 관찰 가능한 결과(로그/콘솔/화면 출력)
4. 이전 단계가 확인된 뒤에 실제 동작으로 전환합니다.
- 한 단계에서 하나의 증거만 제시합니다:
  - “이 함수가 실행됨”
  - “이 이벤트가 발생함”
  - “출력이 변경됨”

## 채팅 위젯 진행 방향 정렬

- 전체 완성형 제품처럼 시작하지 않습니다. 다음으로 보일 수 있는 최소 함수만 추가합니다.
- 인프라보다 먼저 챗봇 동작(입력, 메시지 목록, 렌더링)부터 확인합니다.
- Next.js 결합도는 초기에 낮게 유지하고, 필요 시점에만 확장합니다.
- 함수 추가 시 한 번에 설명:
  - 어떤 입력을 받고
  - 어떤 출력(또는 상태 변경)이 생기며
  - 어떤 엣지 케이스가 있는지
- UI 셸 작업과 동작(이벤트/상태/API/렌더) 작업은 분리합니다.
  - UI 셸: 구조, 배치, 여백, 타이포그래피, 색상 등 시각 요소만
  - 동작: 이벤트 바인딩, 상태, 검색/응답 렌더링 등
- 리팩토링과 기능개발 분리:
  - 리팩토링: 가독성, 이름 정리, 분리, 정리만
  - 기능개발: 사용자에게 보이는 동작 변경만
- provider/search 추상화는 단순 메시지 전송/응답 라운드트립이 검증될 때까지 보류합니다.

## 커밋 승인 흐름

- 커밋은 사용자 승인 후에만 실행합니다.
- 커밋 실행 전:
  - 커밋 메시지 제안
  - 커밋 대상 파일 목록
  - 변경 동작 요약
- 승인 후에 `git add`와 `git commit`을 진행합니다.
